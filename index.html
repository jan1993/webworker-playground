WebWorkers

Projektarbeit im Rahmen der Mastervorlesung "Grundlagen Web-Engineering" am CAS der DHBW

Problematik

JavaScript ist eine Single-Threaded-Umgebnung. Das heißt, die Programme werden immer nur auf einem Thread, dem Hauptthread, ausgeführt. Skripte werden der Reihe nach ausgeführt. Bei großen Berechnungen führt das dazu, dass die Website während der Scriptauführung nicht mehr oder nur noch bedingt reagiert und somit bedient werden kann. Zur Abhilfe wurde von den Webentwicklern deshalb oftmals eine Nebenläufigkeit mittels setTimeout(), setInterval(), XMLHttpRequest und Ereignis-Handlern nachgeahmt. Aber auch damit konnte kein wirkliches Multi-Threading erreicht werden. Seit HTML5 steht nun eine sogenannte WebWorker-API zur Verfügung. Mit dieser kann nun erstmals JavaScript-Code in ein separates File ausgelagert und als eigenständiger Prozess abgearbeitet werden.

Was versteht man unter einem WebWorker? arrow_drop_up

Ein Webworker ist ein Objekt welches JavaScript-Code in einem separaten Thread ausführt, sodass die Leistung des Hauptthreads weiterhin für die Benutzerinteraktion mit der Homepage zur Verfügung stehen kann. Dazu wird beim Erstellen des Objektes auf ein externes JavaScript-File verwiesen. Grundsätzlich unterscheidet man zwei Arten von WebWorkern:

  • Dedicated Worker
  • Shared Worker
  • Service Worker
  • Audio Worker
Während ein Dedicated Worker lediglich aus demjenigen Skript heraus verwendet werden kann das den Worker erzeugt hat, kann ein Shared Worker aus unterschiedlichen Skripten heraus genutzt werden. Bis auf einige Ausnahmen können die meisten JavaScript-Funktionen auch in den Workern verwendet werden.

Wie wird ein WebWorker verwendet? arrow_drop_up

Das Skript welches vom Webworker verarbeitet werden soll muss zunächst in ein separates JavaScript-File ausgelagert werden. Anschließend wird im Hauptfile ein neuer Worker angelegt: var myWorker = new Worker ('myWorker.js')

Beispiel arrow_drop_up

Das nachfolgende Beispiel soll anhand der Berechnung von Primzahlen die Funktionsweise sowie die Vorteile der Web Workers veranschaulichen. Das Beispiel führt die Berechnung einmal ohne und einmal mit Web Workers aus.

Primzahlen werden jeweils bis zur eingebenen Zahl berechnet. Wird hier beispielsweise eine 10 eingetragen, so werden alle Primzahlen von 1-10 berechnet (=> 1 3 5 7 11). Die Funktion zur Berechnung der Primzahlen ist bei beiden Beispielne Identisch. Jedoch wird im Abschnitt "Without Web Workers" die Berechnung im Main Thread durchgeführt. Im Abschnitt "With Web Workers" wird diese Berechnung auf einen Worker ausgelagert und der Input sowie Output zum Main Thread kommuniziert.

Wie wird das Beispiel verwendet?

  • Der Button "Still Responsive?" dient zur Überprüfung ob die Seite noch auf Benutzerinteraktionen reagiert. Sofern die Seite noch reagiert wechselt der Button bei jedem Klick entsprechend die Farbe.
  • Zuerst berechnen wir Primzahlen bis zur Zahl 1000 mit beiden Varianten. Kann man hier bereits einen Unterschied feststellen?
  • Da die Berechnung so schnell durchgeführt wurde konnte man natührlich noch keinen Unterschied feststellen.
  • Im nächsten Schritt steigern wir die Komplexität und berechnen Primzahlen bis zur Zahl 50000. Als erstes wird diese Berechnugn ohne Web Workers durchgeführt. Was können wir nunr feststellen? Bewegt sich der Ladeindikator korrekt? Funktioniert der "Still Responsive Button"?
  • Zum Vergleich wird die selbe Berechnung nun mit Hilfe eines Web Workers durchgeführt. Welche Unterschiede können nun festegestellt werden?
